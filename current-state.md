## 1. What this is
picgraf (a.k.a. pixndx) is a **privacy-preserving photo gallery** with semantic search and network-graph visualization. Privacy-preserving on two counts: (1) all metadata is generated by **open-weight models** (Gemma 3 27B + CLIP ViT-L/14) running in a **controlled AWS VPC** — no third-party AI API ever sees the images; (2) viewing requires an account, and images are served only behind CloudFront signed cookies. Currently deployed as a **single shared gallery** (the owner's personal collection).

## 2. Architecture (as built)
- **Frontend:** React 18 + Vite + TypeScript, Zustand, Tailwind. Visualizations via D3-force / Sigma.js (WebGL) / Graphology; layouts: grid, force-directed network graph (LOD + zoom), color wheel, mood spectrum, clusters, timeline.
- **Backend:** AWS Amplify Gen 2 (CDK). AppSync GraphQL + DynamoDB (`Image`, `ImageRating`, `ManifestUpdate`, `SearchQuery`, vestigial unused `Gallery`). S3 for image sizes / metadata / embeddings / manifest. CloudFront + signed cookies (KeyGroup/OAI). Cognito user pools (email auth, `Admins` group). WAF rate limiting.
- **AI pipeline (the privacy core):** GPU EC2 spot (g5.xlarge) in a VPC; models on EFS; SQS job queue; scales to 0 when idle. `scripts/process_images.py` runs Gemma (description, mood, main_subject, hierarchical tags, colors) + CLIP (768-dim embedding). `preprocessing/cluster_layout.py` computes HDBSCAN clusters, Louvain communities, UMAP 2D layout. Neighbors/edges precomputed for the graph.
- **Deploy:** push to `main` → Amplify build (`tsc && vite build`, strict — unused imports fail). App id `d2lj29cnhp0ir0`. CDN `cdn.picgraf.com` (signed cookies, not usable pre-auth). Pipeline Python: `/usr/bin/python3` (the `~/ml_venv` virtualenv is broken).
- **Key files:** `amplify/data/resource.ts` (schema/auth), `amplify/backend.ts` (infra), `amplify/functions/generateImageCookies/handler.ts` (signed cookies), `src/lib/dataLoader.ts` (CDN manifest load), `src/types/gallery.ts` (types), `scripts/process_images.py` (GPU pipeline).

## 3. Current state / what works
- **Single shared gallery:** every authenticated user sees all images. The line `allow.authenticated().to(['read'])` at `amplify/data/resource.ts:83` is *the* single-gallery assumption.
- Full **ingest → index → visualize → search** loop works. Search is **client-side keyword** over generated metadata (not embedding/semantic search at query time yet).
- Embeddings are **768-dim CLIP only**; `Image.descriptionEmbedding` exists in schema but is never populated.

## 4. Active workstream — Selective-disclosure photo-index agent (DESIGN, not yet built)

### Goal
Evolve from "my single gallery" into a **privacy-preserving photo-index agent**: privately index a collection into a semantic topic space (~1000-dim embedding), and let the owner (modeled as an agent) **selectively disclose** which semantic subsets are visible to which audiences — e.g. "professional" to an employer, a "sanitized" subset to family. Selective-disclosure concepts are transplanted from the `~/dev/exp2` thesis ("faces, not folders" + a resolution dial), applied on the **sell side** (disclosing the index) — *not* to hide the index from an adversary.

### Model as it now stands (evolved through discussion)
- **Two spaces + a map:** (1) a *content topic space* (the ~1000-dim embedding index) where the owner's **faces** live as regions; (2) a *buyer persona/identity space* where requesters are **classified**; (3) the disclosure policy is a map **persona-region → content-face**. "Mom" is a persona region a requester is *discovered* to fall into — not a declared attribute or ACL entry.
- **Buyer-persona discovery is part of the protocol.** Requesters disclose minimal identity; the agent classifies them into a persona, which selects the face. Both parties are exp2-style agents with segmented identities.
- **Disclosure is type-deterministic → a multilevel-security (MLS) / lattice model.** Once a requester is typed (e.g. "mom"), their allowed view is a **fixed materialized sub-index**. Queries only ever filter *within* that view, so within-type querying is **non-interfering** with the excluded complement. This **removes the adaptive-query oracle** — no query stream teaches you about what's excluded.
- **The residual hard problem is in type-space, not query-space.** The only way to learn more is to change type ("type morphing"). The quantity to bound = what the *union of views over all types an entity can reach* reveals. Decomposes into:
  - **(I) Type integrity** — can't forge the evidence the classifier reads to get promoted. *The real home for the crypto: buyer-side identity attestation.*
  - **(II) Sybil / persistence** — can't cheaply mint identities to collect many views. Needs persistent, attested identity + cross-presentation linkage.
  - **(III) Cross-type differential** — a *legitimate* holder of two types learns the boundary between the two views (the classic MLS **inference/aggregation problem**). Largely irreducible; bound / price / accept.
  - **(IV) Morph detection** — link presentations over time, flag deliberate label-promotion probing.
- **Design move that defangs morphing:** separate **hard floors** from **soft faces**. Model truly-sensitive classes (intimate/party photos) as **global excludes** — hidden under *every* reachable type except an explicitly privileged one — not as per-type opt-outs. Morphing within a legit label-lattice then can't surface floored content; the only way in is forging the privileged label = problem (I). Converts "bound all morphing leakage" into "protect a few privileged labels."
- **Deliberate inversion of exp2:** this model **hides that you're hiding** (mom must not know a hidden set exists) — the opposite of exp2's "you cannot hide that you are hiding." Requires **closed-world views**: counts / rankings / "no results" computed over the view only, so "excluded" and "never existed" are indistinguishable.
- **Consequence:** since disclosure is deterministic **set-membership per type** (+ optional sanitization: downscale / blur / EXIF-strip), **QJL / DP / embedding-disclosure largely drop out** — no vectors are ever sent to a buyer. Surviving crypto = **buyer-side identity attestation + Sybil-resistant persistent identity**.

### Roadmap (phased)
- **Phase 0 — schema + space:** add `Collection` / `Face` / `ShareGrant` models; add a text embedder + normalized concatenation to reach ~1000-dim; persist cluster centroids + masses as a coverage distribution.
- **Phase 1 — named-audience disclosure (no heavy crypto):** namespace S3/manifest by collection; remove blanket read (`resource.ts:83`); scoped signed cookies + a **public `redeemShareToken`** pre-auth endpoint; Gemma/CLIP **intent→region** compilation with a preview/confirm step; deterministic per-type views + sanitization.
- **Phase 2 — buyer-persona discovery + type integrity:** attested buyer identity, persona classification, hard-floors, closed-world views.
- **Phase 3 — market crypto (only if pursuing a multi-seller market):** threshold attestation, staking/slashing, Shapley settlement.

## 6. Known issues / loose ends
- **~1000-dim space is aspirational** — pipeline is 768-dim CLIP only; needs a text embedder + normalized/whitened concatenation (block alignment matters for any projection).
- **Gemma metadata is plaintext** — a higher-leakage channel than a vector; must ride the same disclosure policy.
- **GPU running a 27B model is controlled compute, not a TEE** — genuine pipeline-provenance attestation needs confidential-GPU (nascent). CPU-side projection/commitment is attestable now; GPU inference is not.
- **Third-party (depicted-subject) privacy is unaddressed** — the disclosure policy optimizes the owner's interest, not consent of people in the photos. Needs a redaction/consent layer.
- **Signed-cookie revocation is TTL-bounded** (24h today) — true instant revocation needs Lambda@Edge.
